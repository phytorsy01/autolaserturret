01

The decision everything hangs on

Why there are no interrupts

An earlier version drove the servo from a Timer0 interrupt firing every 100 µs. That ISR kept interrupting the HC-SR04 echo measurement on Timer1 mid-count — which is what produced the stuck 5 cm reading. The real fix wasn't a patch. It was deleting interrupts entirely.

So the whole program is one cooperative loop. It runs top to bottom about 50 times a second: poll the button, then either hold still or take one sweep-and-sense step. Because nothing can preempt anything else, nothing can corrupt the echo timing. When you read main.c, there's no ISR to jump to — main() is the entire story.

03

Reading distance — read_cm()

The classic HC-SR04 handshake, but the guards are the interesting part. A 10 µs pulse on TRIG kicks off the burst; then it waits for ECHO to rise, times how long it stays high on Timer1, and converts ticks to centimetres. The two early return 0s are escape hatches so a dead or out-of-range sensor can never freeze the loop.

/* 10 µs trigger, then time the echo pulse on Timer1 */
unsigned int read_cm(void) {
    unsigned int g, ticks;
    unsigned int limit = MAX_RANGE_CM * 72;   // ← range cutoff in ticks

    TRIG = 0; __delay_us(2);
    TRIG = 1; __delay_us(10); TRIG = 0;                // fire the 10 µs ping

    g = 0; while (ECHO == 0) { if (++g > 8000) return 0; }  // echo never came → bail
    TMR1H = 0; TMR1L = 0; TMR1ON = 1;                  // start the clock
    while (ECHO == 1) {
        ticks = (TMR1H << 8) | TMR1L;
        if (ticks > limit) { TMR1ON = 0; return 0; }  // too far → bail
    }
    TMR1ON = 0;
    return ((TMR1H << 8) | TMR1L) / 72;             // ticks → cm
}

The /72, not /58

The usual textbook divisor is 58, which assumes a 1 µs Timer1 tick. You don't have that. At a 20 MHz crystal with Timer1's 1:4 prescaler the tick is shorter, so the correct centimetre divisor is 72. Using 58 here is exactly what skews the reading.

04

Moving the servo without a timer interrupt

A hobby servo wants a pulse every ~20 ms, and the pulse width sets the angle: roughly 1 ms for 0° up to 2 ms for 180°. servo_pulse(us) builds one pulse by hand — drive the pin high, wait us microseconds against Timer1, drive it low. The __delay_ms(16) at the end of each frame spaces those pulses out to roughly the right rate.

The current angle lives in servo_us, scaled between your CLOSE_US (0°) and OPEN_US (180°) endpoints. Crucially, even when stopped the firmware keeps sending the held pulse — that's why the turret freezes stiffly in place instead of going limp.

05

Why it doesn't twitch — target hysteresis

The detect logic is deliberately not a bare if/else. Once a target is seen, it must read clear for several frames in a row — counted by clear_count against CLEAR_FRAMES — before sweeping resumes. One noisy reading can't snap the turret back into sweeping for a single frame and then re-lock. It's the same idea as the button debounce, applied to the target.

if (cm > 0 && cm <= DETECT_CM) { detected = 1; clear_count = 0; }
else if (detected)         { if (++clear_count >= CLEAR_FRAMES) detected = 0; }

07

Things that bite

Servo brownout

The servo is the #1 cause of resets. Give it its own 5 V rail, a shared ground back to the PIC, and a 470 µF cap across its supply. A weak rail makes it judder instead of move.

Echo returns 0 → suspect wiring

When echo reads zero, check ground and the Trig/Echo wiring before touching code. Swap Trig/Echo in wires or code, never both — they cancel out.

Stay off RB6 / RB7

Those carry the PICkit programming signals. Anything wired there fights the programmer.

The /72 divisor

If distances read consistently wrong, confirm it's /72 and Timer1 is on the 1:4 prescaler — not the /58 you'll find in tutorials.

Numbers shown in the simulation above are illustrative. The real values — DETECT_CM, MAX_RANGE_CM, SWEEP_STEP, CLEAR_FRAMES, CLOSE_US, OPEN_US, CENTER_US — are the #defines at the top of your main.c; tune behaviour there.
